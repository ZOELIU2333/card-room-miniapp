# 技术设计：第一版（体验版）

更新时间：2026-06-23

本文档沉淀第一版的技术选型与后端对局核心范围。产品范围见 [PRD](PRD.md)，UI 方向见 [UI 样式要求](UI_STYLE.md)，阶段划分见 [路线图](ROADMAP.md)。

## 1. 第一版定位

- 上线形态：**内部体验版**。用微信小程序体验版二维码在熟人间分发，不走官方审核发布，规避棋牌类目资质问题。
- 后端目标：**做到真实对局**（发牌、出牌、结算、断线重连），不是纯前端 mock。
- 分工：UI 由产品负责人主导（已有 HTML 原型与 Figma 稿）；技术实现先从后端对局核心起，与 UI 并行。

## 2. 技术选型

### 2.1 前端

- 原生微信小程序（WXML / WXSS / TS），不使用 Taro / uni-app。
- 理由：第一版只投微信一端，跨端能力用不上；高级感与材质质感要求对样式掌控最直接；后续 Canvas 牌桌、WebSocket 原生支持最干净。
- 牌桌后续用 Canvas 绘制；材质走实拍图 / 渲染图素材。

### 2.2 后端

- 运行时：**Node.js + TypeScript**。
- 实时通道：**WebSocket**（`ws`，第一版不引入 Socket.IO）。
- 房间状态：**内存为主 + Redis 存活跃房间快照**，支撑断线重连。
- 持久化：**PostgreSQL**，存用户、战绩、积分流水等需要事务的数据。

选型权衡记录：玩家端体验（出牌跟手、不卡、不掉线）主要由架构与前端决定，而非后端语言；棋牌房间并发量小，Node 与 Go 在体验版阶段都远未触及瓶颈。Go 的优势在规模化后的长连接承载与运维成本，体验版用不到。最终选 Node 以最快速度跑通产品闭环。

## 3. 后端架构红线

以下三条从第一版就定死，避免到真实对局阶段返工。

1. **服务端权威**：发牌、洗牌、出牌合法性校验、结算全部在服务端计算。客户端只发送"意图"（如"我要出这几张牌"），不持有也不决定牌局结果。这是防作弊的根本。
2. **单房间串行处理**：一个房间内的所有操作（出牌、加入、退出）串行进入同一个房间实例处理，杜绝并发改状态的竞态。这是棋牌服务的标准做法。
3. **对局逻辑独立成"游戏引擎"模块**：发牌 / 规则 / 结算是纯逻辑，与网络层（WebSocket）解耦，做成纯函数、可单测。换玩法只改这一层。

## 4. 模块边界

| 模块 | 职责 | 依赖 |
|---|---|---|
| 网络层（gateway） | 管理 WebSocket 连接、收发消息、心跳、断线检测 | ws |
| 房间服务（room） | 房间生命周期、成员管理、把玩家意图串行投递给引擎、广播状态 | gateway, engine |
| 游戏引擎（engine） | 纯逻辑：发牌、出牌合法性、回合推进、结算。无网络、无 IO | 无 |
| 状态存储（state） | 活跃房间内存状态 + Redis 快照，支撑断线重连 | redis |
| 持久层（persistence） | 用户、战绩、积分流水 | postgres |

引擎不依赖任何 IO，是整个系统中最先可以写、最容易测的部分。

### 4.1 引擎契约（与玩法无关）

引擎层用一个泛型契约 `GameEngine<S, A, E>` 把玩法规则和外层隔开：

- `createInitialState(playerIds, rng)`：注入随机源产出初始状态（发牌）。
- `step(state, action) => { state, events }`：纯函数状态机。
- `isFinished(state)` / `ranking(state)`：房间层据此收尾、落库。

约定两点为未来玩法预留：动作类型允许一种由房间层计时器触发的"系统/超时"动作（麻将的碰杠胡决策窗口需要，跑得快暂不用）；`ranking` 返回 `{ playerId, rank, score }`，score 含义由各玩法定义，房间/持久层只落库不解释。房间层只依赖此契约，永不认识具体玩法。不预造多玩法框架，待第二个玩法出现再用两个样本提炼公共抽象。

### 4.2 性能与扩展

引擎是纯逻辑层，单次 `step` 为微秒级，永远不是性能瓶颈。真正的性能关键在网络层（长连接承载、广播延迟）与部署架构（多进程/多机按房间分片）。因引擎与网络经契约解耦，未来扩展（水平扩展、甚至把 gateway 换成 Go）不需要改动引擎逻辑。

## 5. 首个玩法

定为**跑得快**：规则相对简单、出牌交互清晰，最适合先把实时对局链路打通。完整玩法库见 PRD 第 4 节，第一版只做这一个玩法的完整闭环。

## 6. 里程碑

- **M1 体验版骨架**：小程序 + 最小 WebSocket 房间服务，跑通一种玩法的完整闭环（建房 → 分享 → 加入 → 发牌 → 出牌 → 结算）。熟人即可真打。
- **M2**：用户体系、积分落库、战绩页、断线重连。
- **M3**：扩展第二、三个玩法（如 240 + 一种麻将），抽象规则引擎。
